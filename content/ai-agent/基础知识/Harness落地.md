## Harness落地

### 🏗️ 第一步：构建知识层 - 让Agent“看得见”

Coding Agent（比如 Claude Code、Codex）的核心问题往往不是“能不能”，而是“知不知道”。因此，首要任务是构建一个清晰的知识层。

- **仓库是唯一的事实来源**：团队的所有讨论、口头约定、架构师的蓝图，如果没写进仓库，Agent就看不见。首先要把**架构决策、分层约束、命名规范**等全部编码进仓库，作为版本化的文件提交。Agent打开项目时就能读取这些文件，就像拥有了整个项目的“世界观”。

- **创建“地图”，而非“手册”**：一个典型误区是写一个巨大的`AGENTS.md`文件，这反而会挤占上下文窗口，让关键信息被稀释。正确的做法是，`AGENTS.md`只做索引和指路（约100行），详细内容按需加载到`docs/`目录中。这能保证Agent的“工作台”始终干净、聚焦。

- **分层构建项目知识**：一个生产级的项目知识库通常包含以下结构：

  text

  ```
  .
  ├── AGENTS.md               # 项目地图，描述项目概况和文档结构（~100行）
  ├── docs/                   # 详细文档目录
  │   ├── architecture.md     # 架构设计文档，描述系统分层、技术选型等
  │   ├── api/                # API文档，详细描述接口定义
  │   ├── guides/             # 开发指南，如环境搭建、部署流程、常见问题
  │   └── decisions/          # 架构决策记录（ADR），记录重要决策的背景和理由
  ├── .cursor/                # 编码助手配置（如Cursor）
  │   └── plans/              # 存储编码计划，实现跨会话任务接续
  ├── .harness/               # Harness配置目录
  │   └── src/                # Harness核心配置源文件
  └── ...                     # 其他项目文件
  ```

  

### 📜 第二步：设计约束与流程 - 给Agent“立规矩”

有了知识，还需要一套明确的“家规”来指导Agent的行为。

- **架构约束：从“告知”到“强制”**：与其在提示词中要求Agent遵循架构规则，不如将其编码为可执行的、自动化的检查。例如，将**分层依赖规则**写入**Linter配置**。当Agent的代码违反规则时，Linter会在提交前自动拦截，并把错误信息反馈给Agent进行修正。
- **使用声明式规范**：将项目规范（如代码风格、安全要求）写入配置文件，让Agent在执行任务前，通过工具来**验证是否符合规范**，而不是依赖它的“常识”或“记忆”。
- **流程约束：采用“规划-执行”模式**：借鉴Cursor等产品的实践，在Agent编写代码前强制要求它先生成一份详细的执行计划（Markdown文件）。这既是给Agent自身的行动指南，也是让人工审核和修正的最后机会。
- **工具约束：坚持“最少工具”原则**：Claude Code的实践非常值得借鉴——它的工具层极其精简，只提供了**Read、Write、Execute、Connect**四种能力原语。其中，`Execute`（即Shell）是核心，它让Agent能够像人类开发者一样，通过组合使用`grep`、`git`、`npm`等通用工具来完成复杂任务。这比给模型配备100个特定工具要强大和灵活得多。

### 🔍 第三步：建立反馈与运行时 - 给Agent“照镜子”

这是确保Agent“不跑偏”的关键。它的核心是建立一个独立、客观的验证系统。

- **建立可观测性**：将日志、指标、追踪系统接入Agent的工作回路，让Agent能主动查询和分析结果（如“服务启动是否在800ms内？”），从而进行自我纠正。
- **引入独立的评估者**：Anthropic的一个关键洞见是，模型不擅长评价自己的工作。最有效的实践是引入一个**独立的Evaluator Agent**来负责验证。它不依赖模型的“直觉”，而是通过**自动化测试**、**静态检查**和**编译检查**等确定性手段来评判工作成果。
- **构建可观测性与反馈闭环**：将任务执行过程中的关键信息（如工具调用、中间结果）通过Tracing系统记录下来，形成闭环反馈，帮助迭代优化Harness的各个组件。
- **管理上下文窗口**：长时任务是Agent的“阿喀琉斯之踵”，模型在窗口快满时会表现出“上下文焦虑”（提前收尾、跳步骤）。一个有效的解法是在窗口将满时，强制Agent将当前进展整理成结构化的**交接材料**（如更新`docs/exec-plans/`中的计划文件），然后**清空上下文**，让下一个会话从交接材料开始，从而彻底解决问题。

### 🚀 第四步：迭代与工具链 - 让Harness“自进化”

Harness本身也需要持续优化。一套工具链能让这个过程更高效。

- **打造自动化优化回路**：可以借鉴LangChain团队的实践。他们用Trace Analyzer Skill自动分析Agent失败的案例，汇总后生成改进建议，再由人工审核后应用到Harness的配置中。他们仅通过优化Harness，就将Agent在编码榜单上的成绩从52.8%提升到了66.5%。
- **选择合适的工具链**：社区已经涌现了许多优秀的开源工具，可以帮助你更快地搭建Harness。
  - **配置管理**：`@madebywild/agent-harness-framework`，提供类似`shadcn`的体验，统一管理多供应商（Claude Code、Copilot等）的Harness配置。
  - **运行时环境**：`@botbotgo/agent-harness`，为多Agent应用提供可操作的运行时层。
  - **状态机支持**：字节开源的Harness通过引入持久化状态机，为长期任务提供可靠的断点续传能力，实时落盘中间状态。
  - **安全控制**：`Ceros`或`Backslash`等工具，提供实时监控、策略执行和审计追踪，实现细粒度的权限控制。
- **模型升级时“做减法”**：Harness的设计应该随着模型能力增强而“变薄”。当新模型表现出更强的上下文理解能力时，应主动移除那些用于弥补旧模型缺陷的组件（如context reset逻辑），保持架构的精简。

**Human Steers. Agents Execute.** 当AI接手具体编码时，工程师的角色正转变为**架构设计师**和**规则制定者**——为AI设计出能稳定、可靠工作的系统
